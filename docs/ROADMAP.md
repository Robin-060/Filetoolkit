# ROADMAP — FileToolkit 全项目实施路线图

> 本文档把 FileToolkit 从零到发布的全过程,拆成 **6 个里程碑、共 38 个可独立交付的步骤**。
> 每个步骤都标注:目标、产出物、验收标准、依赖关系、所在里程碑。
> 本文档为"全景规划",具体实现时每个步骤会另出设计文档(放入 `docs/design/`)。

| 文档版本 | v1.0          |
| 创建日期 | 2026-07-05    |
| 总步骤数 | **38 步**     |

## M1 — MVP 核心功能 ⏳ 待启动

> 目标:实现 4 个高频功能,让产品**首次真正可用**。
> 原则:每个功能独立成 step,可单独验收;先写设计再写代码。

### 通用基础(必须先做)

#### Step 1:统一错误处理与共享类型
- **目标**:建立后端错误模型(`thiserror`)+ 共享类型(`Task`/`Progress`/`TaskStatus`)
- **产出**:`common/error.rs`、`common/types.rs`、设计文档 `design/m1-common.md`
- **验收**:错误可序列化给前端;类型被 commands/pipeline/worker 复用
- **依赖**:M0 完成

#### Step 2:任务队列与进度上报框架
- **目标**:实现 `worker` 模块——多核并行(`rayon`)、进度事件流、失败隔离、可取消
- **产出**:`worker/mod.rs` + 设计文档 `design/m1-worker.md`
- **验收**:
  - 给定 100 个文件,能并行处理且不阻塞 UI
  - 单个失败不中断整批,失败项记录在错误日志
  - 前端能收到实时进度事件(`tauri::Emitter::emit`)
  - 任务可中途取消
- **依赖**:Step 1

#### Step 3:前端基础设施
- **目标**:接入 Pinia 状态管理、Vue Router、UI 组件库(Element Plus / Naive UI)、ESLint + Prettier
- **产出**:`store/task.ts`、路由配置、lint 配置、统一布局组件
- **验收**:`pnpm lint` 通过;有基础布局(侧边栏 + 主区);任务状态可在 store 中流转
- **依赖**:M0 完成

### 四大 MVP 功能(可并行开发,各自一个 step)

#### Step 4:图片批量压缩/转换
- **目标**:支持 JPG/PNG/WebP/GIF/BMP 批量导入,压缩 + 格式转换 + 尺寸调整
- **产出**:`commands/image.rs`、前端 `pages/ImagePage.vue`、设计文档 `design/m1-image.md`
- **技术**:`image` crate 或 `libvips` 绑定
- **验收**:
  - 拖入 50 张图,设定质量参数,批量输出到指定目录
  - 处理前后体积对比可见
  - 失败隔离 + 进度条 + 可取消
- **依赖**:Step 1、2、3

#### Step 5:PDF 合并/拆分/压缩
- **目标**:多 PDF 合并、按页码区间拆分、PDF 压缩
- **产出**:`commands/pdf.rs`、前端 `pages/PdfPage.vue`、设计文档 `design/m1-pdf.md`
- **技术**:`lopdf` / `qpdf` 绑定
- **验收**:
  - 合并 3 个 PDF 为 1 个,页序正确
  - 按页码区间拆分,输出多个文件
  - 压缩后体积下降,内容完整
- **依赖**:Step 1、2、3

#### Step 6:文件批量重命名
- **目标**:模板变量重命名(`{name}` `{date}` `{index:3}` 等),实时预览,冲突检测
- **产出**:`commands/rename.rs`、前端 `pages/RenamePage.vue`、设计文档 `design/m1-rename.md`
- **验收**:
  - 用模板批量改名,执行前**预览新文件名**
  - 检测到重名冲突时阻止执行并提示
  - 支持撤销(优先移入回收站而非直接覆盖)
- **依赖**:Step 1、2、3

#### Step 7:重复文件查重
- **目标**:两阶段查重(大小预筛 → blake3 哈希),分组展示,智能保留策略
- **产出**:`commands/dedup.rs`、前端 `pages/DedupPage.vue`、设计文档 `design/m1-dedup.md`
- **技术**:`blake3` crate
- **验收**:
  - 扫描指定目录,正确识别重复文件并分组
  - 大目录扫描有进度,可取消
  - 支持"保留最新/最大/第一个"策略批量删除
- **依赖**:Step 1、2、3

### M1 收尾

#### Step 8:统一文件选择/拖拽与输出目录组件
- **目标**:把"文件导入 + 输出目录选择 + 处理选项 + 进度展示"抽象成**通用组件**,4 个功能复用
- **产出**:`components/FileDropZone.vue`、`components/TaskProgress.vue`、`composables/useBatchTask.ts`
- **验收**:4 个功能页都用这套组件,行为一致
- **依赖**:Step 4-7(过程中沉淀,边做边抽)

#### Step 9:MVP 集成测试与 Bug 修复
- **目标**:端到端跑通 4 大功能,修复边界问题(超大文件、特殊字符路径、空目录等)
- **产出**:测试用例文档 `design/m1-test-cases.md`,已修复 bug 列表
- **验收**:按测试用例逐项通过;典型边界(1000+ 文件、1GB+ 大文件)不崩溃
- **依赖**:Step 4-8

#### Step 10:M1 发布里程碑
- **目标**:打 tag `v0.1.0-mvp`,更新 README,写发布说明
- **产出**:`CHANGELOG.md`,GitHub Release(若已开仓库)
- **验收**:能从全新环境 clone 后跑起应用
- **依赖**:Step 9

## M2 — 视频/音频处理 + 工程化 ⏸

> 目标:引入 ffmpeg,补齐视频/音频能力;同时补齐工程化(CI/CD、测试)。

#### Step 11:ffmpeg 依赖管理(可选模块机制)
- **目标**:建立"可选外部依赖"机制——首次使用时检测 ffmpeg,缺失则引导下载/定位
- **产出**:`common/dependency.rs`(依赖探测)、ffmpeg 下载脚本、设计文档 `design/m2-deps.md`
- **验收**:
  - 应用启动时探测 ffmpeg 路径(优先 PATH,其次 bundled)
  - 缺失时弹出引导(下载链接或手动指定路径)
  - 探测结果缓存,下次秒开
- **依赖**:M1 完成

#### Step 12:视频剪切/裁剪 + 转码
- **目标**:时间段裁剪、格式转码(MP4/MOV/MKV/WebM)、编码选择(H.264/H.265)
- **产出**:`commands/video.rs`、前端 `pages/VideoPage.vue`、设计文档 `design/m2-video.md`
- **技术**:ffmpeg 子进程调用,解析 stdout/stderr 取进度
- **验收**:
  - 精确剪切时间段,无损模式(`-c copy`)秒级完成
  - 转码输出正确,体积/时长合理
  - 进度条按时间百分比推进
- **依赖**:Step 11

#### Step 13:视频硬件加速 + 压缩 + 音频提取/GIF
- **目标**:NVENC/QSV/VideoToolbox 自动检测;目标码率压缩;音频提取;GIF 生成
- **产出**:扩展 `commands/video.rs`,新增音频/ GIF 命令
- **验收**:
  - 有 N 卡时自动用 NVENC,转码速度显著提升
  - 音频提取输出 mp3/aac,音轨正确
  - GIF 生成可调帧率/尺寸
- **依赖**:Step 12

#### Step 14:音频处理模块
- **目标**:格式转换(MP3/AAC/FLAC/WAV)、剪切合并、音量标准化(`loudnorm`)
- **产出**:`commands/audio.rs`、前端 `pages/AudioPage.vue`
- **验收**:格式转换无损、剪切点精确、标准化后响度统一
- **依赖**:Step 11

#### Step 15:CI/CD 流水线
- **目标**:GitHub Actions 配置——push/PR 时自动跑 lint + cargo test + cargo clippy + 前端 build
- **产出**:`.github/workflows/ci.yml`
- **验收**:PR 合并前所有检查必须绿
- **依赖**:M0(可更早接入,这里集中做)

#### Step 16:自动化测试基线
- **目标**:为核心命令写 Rust 单元测试 + 前端关键流程测试
- **产出**:`src-tauri/tests/`、前端 vitest 用例
- **验收**:核心逻辑测试覆盖率 > 60%,CI 中自动跑
- **依赖**:Step 15

## M3 — 流水线引擎(核心差异化)⏸

> 目标:实现"多操作串联"的灵魂功能,确立差异化护城河。这是本项目区别于所有单一工具的核心。

#### Step 17:流水线数据模型与执行引擎
- **目标**:定义 `Pipeline` / `Node` / `Edge` 类型,实现 DAG 拓扑排序与节点执行调度
- **产出**:`pipeline/model.rs`、`pipeline/executor.rs`、设计文档 `design/m3-pipeline-core.md`
- **验收**:
  - 能解析 JSON 描述的流水线,拓扑排序正确
  - 串行/并行节点都能执行,变量在节点间传递
  - 单节点失败不破坏整体(可配置:停止/跳过)
- **依赖**:M1 完成

#### Step 18:节点类型注册表
- **目标**:把 M1/M2 的所有功能注册为流水线节点(图片压缩、PDF 合并、重命名、查重……)
- **产出**:`pipeline/registry.rs`,各节点的输入/输出 schema
- **验收**:每个 MVP 功能都能作为节点被流水线调用
- **依赖**:Step 17

#### Step 19:流水线前端可视化编辑器
- **目标**:节点式拖拽编排 UI(基于 Vue Flow 或自研)
- **产出**:前端 `pipeline/PipelineEditor.vue`、节点面板、连线交互
- **验收**:
  - 拖拽添加节点、连线、删除
  - 每个节点可配置参数
  - 实时校验:有无环、必填参数是否齐全
- **依赖**:Step 17、18

#### Step 20:流水线模板系统
- **目标**:内置常用模板("照片整理归档"、"合同 PDF 标准化"),支持保存/导入/导出 JSON
- **产出**:模板库(`assets/templates/*.json`)、模板管理 UI
- **验收**:一键加载模板,导出/导入 JSON 正确还原
- **依赖**:Step 19

#### Step 21:Dry-run 预览与执行控制
- **目标**:执行前展示完整处理计划(dry-run),执行中实时进度、暂停/继续/取消
- **产出**:`pipeline/preview.rs`、前端执行控制 UI
- **验收**:
  - dry-run 列出将处理的所有文件及其变换链
  - 执行中可暂停,继续后状态一致
- **依赖**:Step 17、19

#### Step 22:流水线实战打磨
- **目标**:跑通 3-5 个真实场景(照片整理、视频批量压缩、文档标准化),优化体验
- **产出**:场景示例文档、UX 优化清单
- **验收**:真实场景下用户能在 2 分钟内编排并跑完一条流水线
- **依赖**:Step 21

## M4 — 高级功能(可选模块)⏸

> 目标:补齐依赖较重的功能,全部做成**可选模块**(按需下载),保持核心包小。

#### Step 23:Office 文档转换(Doc↔PDF↔HTML)
- **目标**:集成 LibreOffice headless,实现 Office 文档互转
- **产出**:`commands/office.rs`、LibreOffice 可选依赖管理、设计文档 `design/m4-office.md`
- **验收**:Word/Excel/PPT 转 PDF 正确,排版基本保留
- **依赖**:Step 11(依赖管理机制)

#### Step 24:PDF OCR
- **目标**:集成 Tesseract,扫描件/图片 PDF 转**可搜索 PDF**
- **产出**:`commands/ocr.rs`、Tesseract 可选依赖管理
- **验收**:
  - 中英文识别准确率 > 90%(清晰扫描件)
  - 输出双层 PDF(图片层 + 文字层)
- **依赖**:Step 11、Step 23

#### Step 25:批量解压/压缩
- **目标**:zip/7z/rar/tar.gz 批量解压;批量打包(可选密码)
- **产出**:`commands/archive.rs`、前端 `pages/ArchivePage.vue`
- **技术**:`zip` / `flate2` crate;rar 需调用 unrar 二进制
- **验收**:多格式批量解压成功;带密码打包/解压正确
- **依赖**:M1 完成

#### Step 26:文件校验(MD5/SHA)
- **目标**:计算哈希、与预期值比对、批量校验
- **产出**:扩展 `commands/dedup.rs` 或新建 `commands/checksum.rs`
- **验收**:大文件哈希计算有进度;比对结果清晰
- **依赖**:M1 完成

#### Step 27:磁盘占用可视化
- **目标**:扫描目录,按体积生成树状/旭日图
- **产出**:`commands/diskusage.rs`、前端可视化组件
- **验收**:扫描 10GB+ 目录秒级响应;可视化层级清晰
- **依赖**:M1 完成

## M5 — 打磨与发布 ⏸

> 目标:产品化、国际化、打包分发、开源发布。

#### Step 28:国际化(i18n)
- **目标**:接入 vue-i18n,提取所有文案,提供简中/英双语
- **产出**:`locales/zh-CN.json`、`locales/en.json`、i18n 配置
- **验收**:切换语言后所有 UI 文案正确;新增语言只需加 json
- **依赖**:所有功能稳定后

#### Step 29:主题与视觉打磨
- **目标**:统一深/浅色主题,设计应用图标与品牌视觉,统一组件样式
- **产出**:主题变量、图标资源、视觉规范文档
- **验收**:深浅色切换流畅;图标在 Win/Mac 任务栏清晰
- **依赖**:功能稳定后

#### Step 30:打包与分发
- **目标**:Win/Mac/Linux 三端安装包 + **便携版(portable)**
- **产出**:`tauri.conf.json` 打包配置、签名配置、便携版构建脚本
- **验收**:
  - 三端安装包可正常安装运行
  - 便携版免安装可运行
  - 安装包体积合理(核心包 < 30MB,可选模块按需下载)
- **依赖**:Step 28、29

#### Step 31:隐私与安全最终审计
- **目标**:逐一核对 PRD §4.3 的隐私红线(零遥测、本地处理、日志清除等)
- **产出**:隐私审计报告 `design/security-audit.md`
- **验收**:
  - 抓包确认无任何对外网络请求(可选模块下载除外)
  - 日志/临时文件"一键清除"功能可用
  - README 公开声明隐私承诺
- **依赖**:所有功能完成

#### Step 32:文档站与 README
- **目标**:用户文档、功能说明、FAQ、隐私声明
- **产出**:`README.md` 完善、`docs/user-guide/`、可选的 docsify/VitePress 文档站
- **验收**:新用户能照文档独立上手
- **依赖**:功能稳定后

## 优先级与建议节奏

- **强烈建议先彻底完成 M1**(Step 1-10),拿到一个**真正可用的 MVP**,再考虑 M2/M3。
- **M3 流水线是差异化核心**,但要等 M1 把原子功能做扎实后才有意义——否则流水线只是空壳。
- **M2 的 ffmpeg 依赖管理机制(Step 11)是 M4 可选模块的基石**,做完后 M4 的 Office/OCR 能复用同一套机制。
- **CI/CD(Step 15)建议尽早接入**,甚至可以在 M1 中期就做,避免后期债务。
- **隐私审计(Step 31)虽在 M5,但隐私红线贯穿全程**(AGENTS.md §6),开发时就要遵守。
