# AGENTS.md — 项目 AI 协作规范

本文件定义 AI Agent(及人类协作者)在本项目中工作时应遵循的统一规范。
所有 AI Agent 在开始任务前,必须先阅读本文件。

---

## 1. 项目概览

- **项目名称**:FileToolkit(暂定,可调整)
- **定位**:一站式本地文件批量处理工具,主打**本地优先、隐私安全、轻量高效**。
- **技术栈**:Tauri 2 + Rust 后端 + Vue 3(TypeScript)前端。
- **开源策略**:开源。
- **目标平台**:跨平台(Windows / macOS / Linux)。

---

## 2. 文档规范(重要)

> **核心规则:本项目所有文档一律使用 Markdown(.md)格式。**

- ✅ PRD、设计文档、会议纪要、技术调研、API 文档、变更日志、README 等,**全部使用 `.md`**。
- ❌ 不得生成 `.docx` / `.pdf` / `.rtf` / `.txt` 等格式作为项目文档(用户明确要求时除外)。
- 文档统一存放在 `docs/` 目录下,按主题分子目录组织,例如:
  ```
  docs/
  ├── AGENTS.md          # 本文件(AI 协作规范)
  ├── PRD.md             # 产品需求文档
  ├── design/            # 设计文档(架构、模块、数据库等)
  ├── research/          # 技术调研与选型
  ├── meeting/           # 会议纪要
  └── changelog/         # 变更日志
  ```
- 文档命名使用 `kebab-case`(小写 + 连字符),例如 `image-pipeline-design.md`。
- 中文项目,文档正文使用**简体中文**;代码标识符、技术术语保留英文。
- Markdown 内的代码块必须标注语言(如 ` ```rust `、` ```ts `),便于高亮与工具解析。
- 新增/修改文档后,若涉及决策性内容,应在文档顶部或 changelog 中记录变更摘要与日期。

---

## 3. 编码规范

- **Rust 后端**(`src-tauri/`):
  - 遵循 `rustfmt` 默认格式,通过 `cargo fmt` 统一。
  - 通过 `cargo clippy -- -D warnings` 检查,不得有 warning。
  - 公共 API 必须有文档注释 `///`。
  - 错误处理统一使用 `thiserror` + `anyhow`(具体见设计文档)。
- **Web 前端**(`src/`):
  - 使用 TypeScript(禁用纯 JS)。
  - 使用 ESLint + Prettier 统一代码风格。
  - 组件/函数需有简要注释说明用途。
- **命名约定**:
  - Rust:`snake_case`(函数/变量/模块)、`PascalCase`(类型/结构体)。
  - TS:`camelCase`(变量/函数)、`PascalCase`(类/组件/类型)。
  - 文件名:Rust 用 `snake_case.rs`,前端用 `kebab-case.ts`/`PascalCase.tsx`。

## 4. 项目结构约定

```
file-toolkit/
├── docs/                # 所有文档(.md)
├── src-tauri/           # Rust 后端
│   ├── src/
│   │   ├── commands/    # 暴露给前端的 Tauri 命令(每功能一模块)
│   │   │   ├── image.rs
│   │   │   ├── pdf.rs
│   │   │   ├── video.rs
│   │   │   ├── rename.rs
│   │   │   └── dedup.rs
│   │   ├── pipeline/    # ★ 流水线引擎(串联多个操作)
│   │   └── worker/      # 任务队列 + 进度上报
│   ├── Cargo.toml
│   └── tauri.conf.json
├── src/                 # Web 前端
│   ├── pages/
│   ├── pipeline/        # 流水线可视化编辑器
│   └── store/
├── scripts/             # 构建/发布脚本
└── README.md
```

- 每个功能模块在 `commands/` 下独立成文件,职责单一。
- **流水线引擎(`pipeline/`)** 是核心差异化能力,单独成模块,详见设计文档。
- 跨模块共享的类型/工具放在 `common/` 或 `utils/`。

## 5. 工作流程规范

### 任务开始前
1. 阅读本文件(`AGENTS.md`)与 `PRD.md`。
2. 通过 `TodoWrite` 列出任务清单,优先级标注清晰。
3. 涉及架构/功能决策时,优先在 `docs/design/` 沉淀设计文档再编码。

### 任务进行中
4. 一次只做一件事,频繁小步提交(用户要求时才 commit/push)。
5. 修改既有代码前先阅读相关文件,保持既有风格一致。
6. 文件/批量操作类功能,必须保证:**失败隔离 + 可撤销 + 可取消**(见 PRD)。
7. 涉及外部二进制依赖(ffmpeg / LibreOffice / Tesseract)时,做成**可选模块**,按需下载。

### 任务完成后
8. 运行检查:`cargo fmt && cargo clippy && cargo test`(Rust)、前端 lint。
9. 更新相关文档(架构/接口/变更日志)。
10. 在回复中如实汇报:做了什么、是否验证、有无遗留问题。

## 6. 安全与隐私红线

本项目主打**本地隐私安全**,以下为不可违反的原则:

- 🔒 **任何文件内容不得上传到网络**。所有处理在本地完成。
- 🔒 **默认零遥测**。如需收集崩溃日志,必须显式征得用户同意,且可在设置中一键关闭。
- 🔒 处理日志、临时文件默认存放于用户私有目录,提供"一键清除"功能。
- 🔒 不引入任何强制联网的依赖;核心功能必须离线可用。
- 🔒 涉及 EXIF、文件元数据时,提供"去除隐私信息"选项。

- 回复使用简体中文。
- 汇报进度时区分:**已完成 / 进行中 / 待确认**。
- 遇到需求模糊或多种方案,先用 `AskUserQuestion` 与用户确认,不要擅自决定重大方向。
